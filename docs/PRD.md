# PRD: MVP системы контроля обязательств и эскалаций

## Назначение документа

Документ описывает MVP системы, которая помогает фиксировать обязательства по людям, автоматически напоминать о них, проверять результат, эскалировать просрочки и ставить оператору задачи на ручное воздействие, включая звонок. Система проектируется как движок обязательств и эскалаций с хранением состояния в PostgreSQL и оркестрацией через n8n.[1][2]

## Контекст и проблема

В организации существует большое количество договоренностей и поручений, по которым исполнители избегают ответственности, не дают результат в срок, не отвечают на сообщения или постоянно требуют ручных follow-up действий. В текущем процессе оператору приходится вручную писать письма, отправлять напоминания, контролировать ответы, отдельно инициировать звонки и вручную помнить, кого и по какому поводу нужно дожимать.

Проблема особенно выражена в сценариях массовых однотипных обязательств, когда одна и та же задача назначается большому числу людей, например заполнение отчетности. В таких кейсах ручной контроль не масштабируется, а отсутствие единой системы состояния приводит к потере фокуса и запоздалым эскалациям.[3]

## Цель MVP

Создать минимально жизнеспособную систему, которая позволяет:

- заводить индивидуальные и массовые обязательства;
- автоматически отправлять первичные уведомления и follow-up напоминания;
- работать с Telegram через клиентский API (MTProto), а не через Telegram Bot API, поэтому без зависимости от bot-кнопок как основного UX-паттерна;[4][5][6]
- отправлять и получать email через Microsoft Exchange, используя SMTP/IMAP и при необходимости EWS-подходы через HTTP;[7][8]
- анализировать свободные слоты в календаре и назначать встречи (meeting invite) как действие эскалации;
- хранить у каждого человека номер телефона и автоматически создавать оператору задачу «позвонить», если текстовые каналы не дали результата;
- отслеживать факт выполнения обязательства;
- запускать эскалации по правилам;
- формировать ежедневный фокус оператора.

## Бизнес-результат

MVP должен снизить объем ручных напоминаний и сделать контроль обязательств воспроизводимым. Ожидаемый эффект — уменьшение количества задач, которые держатся только в голове оператора, повышение доли задач, доведенных до результата без ручного микроконтроля, и более раннее выявление случаев, где нужен звонок, встреча или эскалация руководителю.[9][10]

## Продуктовая гипотеза

Если каждое обязательство хранится как отдельная сущность со статусом, дедлайном, историей касаний, правилом проверки результата и следующем действием, то значительная часть контроля может быть автоматизирована. В этом случае оператор работает не как “человек-память”, а как владелец очереди исключений и сложных случаев.[3][11][12]

## Границы MVP

### Что входит в MVP

- Создание шаблонов задач.
- Создание массовых кампаний на основе шаблонов.
- Создание персональных assignment-записей на каждого человека.
- Хранение людей, каналов связи и номера телефона.
- Telegram-канал через client API / MTProto adapter, без опоры на Telegram Bot API.[4][5][6]
- Email-канал через Exchange-интеграцию, практически — SMTP для отправки, IMAP для приема ответов, при необходимости EWS через HTTP Request.[7][8]
- Календарь рабочих/нерабочих дней для планирования касаний и соблюдения quiet period.
- Интеграция с календарем для поиска свободных слотов и назначения встреч.
- Планировщик напоминаний через `Schedule Trigger` в n8n.[13]
- История касаний по каждому обязательству.
- Базовые статусы и эскалации.
- Постановка операторской задачи на звонок по номеру телефона.
- Daily digest оператору.
- Базовая или полуавтоматическая проверка результата.

### Что не входит в MVP

- Полноценное веб-приложение с большим пользовательским интерфейсом.
- Интеграция с телефонией или автодозвоном.
- Сложный NLP-анализ свободного текста.
- Mobile app.
- Сложная ролевка и многоуровневый RBAC.
- Многоканальная коммуникация beyond Telegram client + Exchange email + операторский звонок.
- Полноценный SLA/процессный движок enterprise-уровня.

## Основные пользователи

| Роль | Описание |
|---|---|
| Оператор контроля | Создает задачи, запускает массовые кампании, получает сводки, разбирает просрочки, совершает звонки |
| Исполнитель | Получает обязательства, отвечает по ним, сообщает дату, готовность или блокер |
| Руководитель исполнителя | Получает эскалации, если задача просрочена или исполнитель не реагирует |
| Администратор | Настраивает шаблоны, каналы, политики эскалации и системные справочники |

## Ключевые сущности продукта

### Person

Карточка человека, которому можно ставить обязательства.

Обязательные поля MVP:

- `full_name`
- `phone_number`
- `email` или `exchange_email`
- `telegram_peer_id` или иной идентификатор для client API
- `manager_person_id`
- `role_code`
- `is_active`

Критически важно, чтобы номер телефона был обязательным полем, так как часть просрочек должна переводиться в ручное действие “позвонить”.

### Task Template

Шаблон обязательства, например “Заполнить отчетность”, с типовым текстом, дефолтным дедлайном, типом проверки результата и политикой эскалации.

### Project

Системный проект (workspace/project space) как контейнер, в рамках которого ведутся индивидуальные и массовые задачи.  
Пример: системный проект `Контрольные точки`, внутри которого оператор ведет задачи по множеству внутренних проектов компании.

Минимальные поля MVP:

- `project_code`
- `project_name`
- `owner_person_id`
- `status` (`active` / `paused` / `archived`)
- `start_date`
- `target_date`

### Target Object (проект из Excel)

Бизнес-объект из Excel, для которого создаются задачи внутри системного проекта.  
Это не системный `Project`, а отдельная сущность предметной области (например, внутренний проект компании для перевода на новый инструмент).

Минимальные поля MVP:

- `target_object_external_key` (детерминирующий ключ из Excel)
- `target_object_name`
- `responsible_person_ref`
- `source_import_version`
- `source_payload_snapshot`

### Task Batch

Массовая кампания на основе шаблона в рамках конкретного проекта, например “Отчетность за апрель 2026”, которая порождает много assignment-записей.

### Task Assignment

Персональный экземпляр обязательства для конкретного человека. Это основная рабочая сущность системы. Один шаблон и одна кампания могут порождать множество assignment, и каждый assignment живет отдельно по статусу, дедлайну, касаниям и эскалациям.[3]

### Touchpoint

Любое взаимодействие по задаче: сообщение в Telegram, письмо, входящий ответ, эскалация, постановка задачи на звонок, подтверждение результата.

### Evidence

Доказательство выполнения: ответ из таблицы, входящее событие, ручное подтверждение, запись из внешней системы.

### Operator Queue Item

Операторская задача на ручное действие: звонок, встреча, ручной разбор.

### Calendar Event

Событие встречи, создаваемое системой в календаре при эскалации. Содержит организатора, участников, слот, тему, комментарий по задаче и ссылку на assignment.

## Основной сценарий использования

1. Оператор создает шаблон задачи или выбирает существующий.
2. Оператор создает индивидуальное обязательство или массовый batch.
3. Система создает персональные assignment-записи.
4. Система отправляет первичное уведомление по Telegram client API и/или Exchange email.[4][7]
5. Система по расписанию проверяет, у каких задач наступило следующее действие, и отправляет reminder.[13][12]
6. Если пользователь отвечает, система обновляет статус обязательства.
7. Если пользователь сообщает о выполнении, система запускает проверку результата.
8. Если результата нет и просрочка наступила, система повышает уровень эскалации.
9. Если текстовые каналы не сработали, система создает операторскую задачу “позвонить”.
10. Оператор получает ежедневную сводку с фокусом на проблемные обязательства.

## Функциональные требования

### FR-1. Управление людьми

Система должна позволять хранить карточки людей с именем, email, Telegram-идентификатором, номером телефона, ролью и руководителем.

#### Критерии приемки

- Нельзя создать активного человека без номера телефона.
- Нельзя создать активного человека без хотя бы одного текстового канала связи.
- Можно указать руководителя для будущих эскалаций.

### FR-2. Шаблоны обязательств

Система должна позволять создавать и редактировать шаблоны задач.

Поля шаблона:

- название;
- текст уведомления;
- текст напоминания;
- дефолтный срок;
- тип проверки результата;
- escalation policy;
- приоритет;
- разрешенные каналы.

#### Критерии приемки

- Шаблон можно создать без запуска задачи.
- Шаблон можно использовать для индивидуальных и массовых назначений.

### FR-3. Массовые кампании

Система должна позволять запускать batch-задачу по выбранной аудитории в рамках проекта, создавая отдельный assignment на каждого участника.[3]

#### Критерии приемки

- Одна массовая кампания может породить N независимых assignment.
- Статус каждого assignment ведется отдельно.
- Ошибка по одному assignment не должна ломать batch целиком.
- Batch нельзя создать без `project_id`.

### FR-4. Индивидуальные обязательства

Система должна позволять создать обязательство для одного человека с конкретным дедлайном и политикой действий в рамках проекта.

#### Критерии приемки

- После создания assignment получает уникальный `task_code`.
- Для assignment рассчитывается `next_action_at`.
- Assignment нельзя создать без `project_id`.

### FR-5. Telegram через client API

Система должна уметь отправлять и принимать сообщения в Telegram через клиентский API / MTProto adapter, а не через Bot API.[4][5][6]

#### Ограничения MVP

- Не использовать bot inline keyboard как обязательную часть UX.
- Основной формат ответа пользователя — короткие текстовые команды или структурированный ответ.
- Допускается fallback на ссылку в web-form, но не как обязательный первый вариант.

#### Критерии приемки

- Система умеет отправить исходящее сообщение в Telegram клиентом.
- Система умеет принять входящее сообщение и привязать его к assignment.
- Система умеет распарсить минимум такие команды: `OK`, `DATE`, `DONE`, `BLOCK`, `CALL`.

### FR-6. Email через Exchange

Система должна уметь отправлять письма через Exchange-контур и принимать входящие ответы из почты. Для MVP допускается реализация через SMTP на отправку и IMAP на чтение, а при необходимости — EWS через HTTP Request.[7][8]

#### Критерии приемки

- Система умеет отправить email по assignment.
- Система умеет получить входящий reply и попытаться распознать `task_code`.
- Ошибки доставки логируются.

### FR-7. Планировщик напоминаний

Система должна по расписанию выбирать задачи, у которых наступил срок следующего действия, и выполнять нужный action через n8n `Schedule Trigger` и выборку из PostgreSQL.[13][2]

#### Критерии приемки

- Scheduler запускается автоматически.
- Если у задачи `next_action_at <= now()`, она попадает в обработку.
- После выполнения действия у задачи обновляется `next_action_at`.

### FR-8. Прием ответов и смена статусов

Система должна обновлять assignment на основе входящих ответов из Telegram или email.

Поддерживаемые статусы MVP:

- `new`
- `notified`
- `acknowledged`
- `in_progress`
- `done_pending_check`
- `done`
- `overdue`
- `escalated`
- `blocked`
- `cancelled`

#### Критерии приемки

- Ответ `OK` переводит assignment в `acknowledged` или аналогичный статус.
- Ответ с датой обновляет ожидаемую дату и переносит следующее действие.
- Ответ `DONE` переводит assignment в `done_pending_check`.
- Ответ `BLOCK` переводит assignment в `blocked`.

### FR-9. Проверка результата

Система должна поддерживать минимум три способа проверки результата:

- ручное подтверждение;
- наличие записи во внешней таблице;
- входящее событие от внешней системы через webhook.

Метод проверки результата определяется отдельно:

- для каждой индивидуальной задачи (`assignment`) при необходимости;
- для шаблона/блока однотипных задач (`task template` / `task batch`) по умолчанию.

Для каждого метода допускается и ожидается реализация отдельного `sub-workflow` в n8n (например, SQL checker, API checker, Excel ingestion checker), который вызывается из общего `Result checker workflow`.

#### Критерии приемки

- Assignment нельзя безусловно считать завершенным, если policy требует проверки результата.
- После успешной проверки создается `evidence`, а статус становится `done`.

### FR-10. Эскалации

Система должна запускать эскалации, если просрочен дедлайн или превышено количество касаний без результата.[9][10][11]

Минимальные виды эскалации:

- повторное текстовое напоминание;
- письмо с более жесткой формулировкой;
- письмо/уведомление руководителю;
- задача оператору на звонок;
- предложение и назначение встречи по свободным слотам календаря;
- операторская задача на ручной разбор.

#### Критерии приемки

- У assignment растет `escalation_level`.
- Каждая эскалация фиксируется как touchpoint.
- При определенных условиях создается operator queue item.

### FR-11. Задача на звонок

Система должна автоматически создавать операторскую задачу “позвонить”, если политика эскалации требует ручного контакта и у контакта есть номер телефона.[14]

#### Критерии приемки

- Создается queue item с типом `call`.
- В queue item передается номер телефона.
- В queue item передается короткий контекст: кто, по какой задаче, какой дедлайн, сколько было касаний.
- Если номера телефона нет, создается задача на исправление качества данных.

### FR-12. Daily digest

Система должна раз в день отправлять оператору краткую сводку.

Сводка минимум включает:

- просроченные задачи;
- задачи без реакции после нескольких касаний;
- новые call-task;
- эскалации на руководителей;
- задачи в `done_pending_check`.

#### Критерии приемки

- Digest отправляется автоматически.
- Digest содержит ссылки или идентификаторы задач для быстрого перехода в разбор.

### FR-13. Календарь рабочих и нерабочих дней

Система должна учитывать календарь рабочих/нерабочих дней при планировании напоминаний, эскалаций и операторских действий.

Минимальные требования MVP:

- поддержка выходных (суббота, воскресенье);
- поддержка праздничных дней из настраиваемого календаря;
- пересчет `next_action_at` на ближайшее рабочее время при попадании в нерабочий период;
- возможность задавать рабочий календарь и quiet period в настройках policy.

#### Критерии приемки

- Если `next_action_at` попадает на выходной/праздник, действие переносится на ближайший рабочий слот.
- Напоминания и эскалации не отправляются в нерабочие дни.
- Правила календаря применяются одинаково для Telegram, email и операторских задач.

### FR-14. Назначение встреч по свободным слотам календаря

Система должна уметь анализировать свободные слоты календарей участников и назначать встречу как эскалационное действие.

Минимальные требования MVP:

- интеграция с календарем Exchange для чтения занятости (availability/free-busy);
- поиск минимум одного общего свободного слота между обязательными участниками;
- создание calendar invite с темой, временем, участниками и ссылкой на `task_code`;
- запись факта назначения встречи в `touchpoint` и привязка к assignment;
- fallback: если общий слот не найден в заданном окне, создается operator queue item типа `meeting_manual`.

#### Критерии приемки

- Система может подобрать общий слот в окне поиска (например, ближайшие 5 рабочих дней).
- Система может создать встречу в календаре и разослать приглашения участникам.
- В assignment фиксируются `meeting_event_id`, `meeting_start_at`, `meeting_status`.
- При отмене/переносе встречи изменение отражается в touchpoint-истории.

#### Бизнес-параметры алгоритма слотов (MVP)

Параметры ниже должны быть настраиваемыми через policy шаблона задачи, с дефолтами на уровне системы:

- `meeting_duration_min`: длительность встречи в минутах (дефолт 30).
- `search_window_workdays`: окно поиска в рабочих днях от текущего времени (дефолт 5).
- `workday_start_local` / `workday_end_local`: рабочие часы для подбора слотов (дефолт 10:00-18:00).
- `timezone`: таймзона расчета и создания встречи (дефолт `Europe/Moscow`).
- `required_attendees`: обязательные участники (минимум исполнитель + оператор или владелец процесса).
- `optional_attendees`: опциональные участники (например, руководитель).
- `slot_step_min`: шаг сетки слотов в минутах (дефолт 15).
- `max_slot_candidates`: максимум альтернатив, которые система готовит для отправки/выбора (дефолт 3).
- `slot_selection_strategy`: стратегия выбора слота, минимум поддержка `first_best`.

Определение стратегии `first_best` для MVP:

- система сортирует кандидатные слоты по времени возрастанию;
- выбирает первый слот, в котором свободны все `required_attendees`;
- при равных условиях предпочитает слот в тот же рабочий день;
- если подходящий слот не найден, создается `meeting_manual` в operator queue.

### FR-15. Проверка результата через запрос во внешние системы (SQL/API)

Система должна уметь проверять выполнение обязательства через параметризуемые запросы во внешние системы.

Минимальные требования MVP:

- поддержка двух технических режимов: `sql_query` и `http_api`;
- настройка метода на уровне шаблона/блока задач с возможностью переопределения на assignment;
- хранение параметров проверки в `verification_policy` (например: `system_code`, `query_or_endpoint`, `params_mapping`, `success_condition`, `timeout_sec`, `retry_policy`);
- выполнение проверки через выделенный `sub-workflow` с идемпотентным запуском;
- запись результата проверки в `evidence` с техническим и бизнес-результатом.

#### Критерии приемки

- Для задачи можно указать `verification_method = sql_query` или `verification_method = http_api`.
- Проверка запускается автоматически при переходе в `done_pending_check`.
- При выполнении `success_condition` статус становится `done`.
- При технической ошибке после исчерпания retry создается operator queue item типа `verification_manual`.

### FR-16. Проверка результата через загрузку Excel/CSV

Система должна поддерживать проверку результата на основе загруженного файла (Excel/CSV) как отдельный метод верификации.

Минимальные требования MVP:

- поддержка форматов `.xlsx`, `.xls`, `.csv`;
- настройка шаблона файла и обязательных колонок на уровне policy;
- сопоставление строк файла с assignment по `task_code` или иному внешнему ключу;
- валидация структуры и логирование ошибок на уровне строки;
- выполнение загрузки и валидации через отдельный `sub-workflow` (`file_ingestion_verification`).

#### Критерии приемки

- Система принимает файл в допустимом формате и запускает проверку.
- Для валидных строк создается/обновляется `evidence`.
- Ошибочные строки попадают в отчет ошибок с причиной.
- Если файл невалиден целиком, создается operator queue item типа `verification_manual`.

### FR-17. Персонализированные рассылки из Excel/CSV (mail merge / grouped digest)

Система должна поддерживать массовую персонализированную рассылку, где контент письма/сообщения формируется на основе данных из единого файла и группируется по получателю.

Минимальные требования MVP:

- загрузка единого Excel/CSV-файла с записями (например, список проектов и ответственных);
- разбор файла через отдельный `campaign_personalization` sub-workflow;
- группировка строк по получателю (например, один ответственный -> одно письмо со списком его проектов);
- рендер персонализированных сообщений из шаблона с плейсхолдерами;
- режим `Preview & Edit` до запуска отправки: просмотр всех сообщений, ручная правка отдельных сообщений, исключение получателей;
- двухшаговый запуск: `Generate draft` -> `Approve and Send`;
- запуск рассылки только после явного подтверждения оператором.
- approval второго лица не требуется (single-operator approval).
- после `Approve and Send` разрешена ручная правка сообщений; изменения применяются только к еще не отправленным сообщениям и фиксируются в audit trail.
- при изменении исходного Excel после генерации draft выполняется пересборка персонализации с потерей ручных правок.
- import policy для Excel обязателен и включает: `import_version`, `imported_at`, `imported_by`.
- внешний детерминирующий ключ целевого объекта в Excel: специальное поле `target_object_external_key`.
- при конфликте данных между Excel и внутренней БД в рамках кампании источником истины считается Excel.
- если проект отсутствует в новой загрузке Excel, assignment не закрывается автоматически; система создает предупреждение оператору.
- если owner из Excel не найден в справочнике людей, создается запись в review-очередь на дополнение справочника (автосоздание отключено).
- фиксируется флаг отправки письма `email_sent_flag`: автоматически при успешной отправке каналом, вручную оператором при fallback-отправке через внешний клиент.

#### Критерии приемки

- После загрузки файла создается черновик кампании со статусом `ready_for_review`.
- Для каждого получателя сформирован отдельный персонализированный draft message.
- Оператор может отредактировать тему/текст конкретного сообщения до отправки.
- После подтверждения запускается отправка и для каждого сообщения фиксируется `sent`/`failed` и причина ошибки.
- Все отправленные сообщения привязаны к соответствующим assignment/batch и проекту.
- Для каждой загрузки файла создается `import_version`, а в журнале импорта сохраняются `imported_at` и `imported_by`.
- Для нераспознанных owner создаются review-записи, блокирующие автоотправку по таким строкам до решения оператора.
- При ручной fallback-отправке оператор может выставить `email_sent_flag` вручную с комментарием причины.

### FR-18. Проектный контекст задач и кампаний

Система должна вести индивидуальные задачи и массовые кампании в рамках проекта как обязательного бизнес-контекста.

Минимальные требования MVP:

- каждая сущность `task_assignment` содержит `project_id`;
- каждая сущность `task_batch` содержит `project_id`;
- один assignment может относиться только к одному проекту (strict 1:1);
- пользователь может фильтровать assignment и batch по проекту;
- в карточке assignment отображается проект и ссылка на карточку проекта;
- в массовой персонализированной кампании все сообщения и generated assignments наследуют `project_id`.
- каждый assignment дополнительно содержит ссылку на целевой объект из Excel (`target_object_id`) внутри выбранного системного проекта;
- поддерживается фиксация промежуточного прогресса по задаче в формате:
  - `% completion`;
  - `progress_note`;
  - `next_commitment_date`;
- обновление дедлайна при частичном прогрессе выполняется только вручную оператором.

#### Критерии приемки

- Нельзя создать assignment без `project_id`.
- Нельзя создать batch без `project_id`.
- Нельзя создать assignment по строке Excel без `target_object_external_key`.
- В UI доступны фильтры по проекту в списках assignment и batch.
- В UI доступны фильтры по `target_object` внутри системного проекта.
- В отчетах и KPI доступен срез по проекту.

## Нефункциональные требования

### Архитектура

- Состояние системы хранится в PostgreSQL, а не только внутри execution n8n.[1][2]
- Backend-API реализуется как отдельный сервис между UI и n8n/DB.
- Владелец state machine: backend + БД (n8n не является источником истины по статусам).
- n8n используется как orchestration-слой: scheduler, channel routing, inbound/outbound обработка, проверки и эскалации.[1][12]
- Для масштабирования должна быть предусмотрена возможность queue mode с Redis и Postgres.[1]

### Производительность

- Планировщик должен обрабатывать due actions не реже одного раза в 10–15 минут.
- Массовый batch на сотни пользователей не должен требовать по одному долгоживущему `Wait` execution на человека; выбор действий должен происходить по scheduler-паттерну.[1][12]

### Надежность

- Все исходящие доставки и ошибки логируются.
- Workflow должны быть идемпотентны.
- Повторный запуск не должен плодить дубли operator queue или duplicate assignment при одинаковом external key.
- Консистентность допускается eventual consistency в окне 1-5 минут.
- Для Telegram/Email/API применяется retry policy: максимум 3 попытки с backoff.

### Наблюдаемость

- Для каждого assignment должна быть доступна история touchpoints.
- Должно быть видно текущее состояние, следующий шаг, дедлайн, число касаний и уровень эскалации.

### Безопасность

- Доступ к данным и credential должен быть ограничен.
- Сессия Telegram client API должна храниться защищенно.
- Почтовые учетные данные Exchange должны храниться в credential store n8n или внешнем секрет-хранилище.

## Бизнес-правила

### BR-1

Одно обязательство на человека — это отдельный assignment, даже если задача массовая.[3]

### BR-2

Ответ “принял” не считается результатом.

### BR-3

Ответ “готово” не всегда считается окончательным завершением; при required check задача переходит в `done_pending_check`.[11]

### BR-4

Каждое исходящее и входящее взаимодействие фиксируется как touchpoint.

### BR-5

Если текстовые каналы не сработали, система должна уметь перевести обязательство в ручное воздействие — звонок или разбор.

### BR-6

Если для человека отсутствует номер телефона, система должна сигнализировать о проблеме качества данных.

### BR-7

Индивидуальные и массовые задачи создаются и ведутся только в рамках конкретного проекта.

### BR-8

Персонализированная массовая рассылка должна проходить этап предпросмотра и ручного подтверждения перед отправкой.

### BR-9

В спорных результатах верификации (противоречивые данные из SQL/API/file) финальное решение принимает оператор.

### BR-10

Календарная встреча в рамках эскалации создается от имени оператора.

### BR-11

Статус `cannot_be_done` может быть установлен только оператором и только с обязательным комментарием причины.

### BR-12

Термин "проект" в системе разделяется на два уровня:

- `Project` — системный контейнер для организации задач и наборов задач;
- `Target Object` — проект/объект из Excel, по которому фактически выполняются действия и создаются assignment.

## Метрики успеха MVP

Система считается успешной, если по итогам пилота наблюдается:

- снижение количества ручных напоминаний на одну задачу;
- рост доли задач, закрытых без ручного дожима;
- уменьшение числа забытых и потерянных обязательств;
- прозрачность причин, по которым задача не выполнена;
- предсказуемое появление call-task вместо ручного хаоса;
- наличие понятной очереди проблемных кейсов для оператора.

### KPI-таблица пилота (baseline/target/формула)

| KPI | Формула | Baseline | Target MVP | Период | Источник |
|---|---|---:|---:|---|---|
| Доля задач, доведенных до итогового исхода | `(done + cannot_be_done) / created` по assignment, где `cannot_be_done` фиксируется как отмена с валидной причиной невыполнимости | 0% (текущая экспертная оценка: задачи не дожимаются) | >= 50% | еженедельно + итог пилота | PostgreSQL (`task_assignment`) |
| Доля задач, закрытых без звонка | `done_without_call / done_total` | TBD | TBD | еженедельно | `touchpoint`, `operator_queue_item` |
| Среднее время до первого ответа | `AVG(first_response_at - first_outbound_at)` | TBD | TBD | еженедельно | `touchpoint` |
| Доля задач с просрочкой | `overdue_count / created` | TBD | TBD | еженедельно | `task_assignment` |
| Доля задач, переведенных в call-task | `call_task_count / created` | TBD | TBD | еженедельно | `operator_queue_item` |
| Полезность эскалации | `% задач, где после эскалации получен ответ или зафиксирован прогресс в течение N дней` | TBD | TBD | еженедельно | `touchpoint`, `task_assignment` |

Примечание по расчету времени исхода:

- KPI по времени считается от `assignment.created_at` до момента первого финального исхода: `done_at` или `cannot_be_done_at`.

## Матрица статусов и переходов (MVP)

Ниже зафиксирована полная матрица для статусов из FR-8. Переходы, не перечисленные в таблице как допустимые, считаются недопустимыми.

### Допустимые переходы

| Текущий статус | Событие | Условие | Следующий статус | Инициатор |
|---|---|---|---|---|
| `new` | отправлено первичное уведомление | доставка успешна | `notified` | система |
| `notified` | получен ответ `OK` | ответ сопоставлен с `task_code` | `acknowledged` | исполнитель |
| `notified` | получен ответ `DATE` | дата валидна и сохранена | `in_progress` | исполнитель |
| `acknowledged` | получен ответ `DATE` | дата валидна и сохранена | `in_progress` | исполнитель |
| `in_progress` | получен ответ `DONE` | проверка требуется policy | `done_pending_check` | исполнитель |
| `acknowledged` | получен ответ `DONE` | проверка требуется policy | `done_pending_check` | исполнитель |
| `notified` | получен ответ `DONE` | проверка требуется policy | `done_pending_check` | исполнитель |
| `done_pending_check` | проверка результата успешна | evidence создан | `done` | система/оператор |
| `done_pending_check` | проверка результата неуспешна | evidence не подтвержден | `in_progress` | система/оператор |
| `new`/`notified`/`acknowledged`/`in_progress` | дедлайн просрочен | `now() > deadline_at` | `overdue` | система |
| `overdue` | выполнен шаг эскалации | сработало правило policy | `escalated` | система |
| `escalated` | очередной шаг эскалации | `escalation_level` увеличен | `escalated` | система |
| `new`/`notified`/`acknowledged`/`in_progress`/`overdue`/`escalated` | получен ответ `BLOCK` | причина блокировки принята | `blocked` | исполнитель |
| `blocked` | unblock оператором | зафиксирован новый план/дата | `in_progress` | оператор |
| `new`/`notified`/`acknowledged`/`in_progress`/`blocked`/`overdue`/`escalated` | ручная отмена | валидная причина отмены | `cancelled` | оператор/админ |

### Недопустимые переходы

- Любой переход **из** `done` в любой другой статус.
- Любой переход **из** `cancelled` в любой другой статус.
- Переход в `done` без промежуточного `done_pending_check`, если в policy требуется проверка.
- Переход `new -> done`, `new -> in_progress`, `new -> escalated` без факта уведомления/системного триггера.
- Переход `blocked -> done` без повторной фиксации выполнения и проверки.
- Переход `notified -> done` в обход `done_pending_check`, если в policy включен check.

### Правила изменения `escalation_level`

- Уровень растет на 1 при каждом шаге эскалации.
- Для `escalated` допускается повторный self-transition `escalated -> escalated` при росте уровня.
- Все изменения уровня фиксируются как `touchpoint`.

## Ограничения и риски

### Риск 1. Telegram client API

Интеграция через MTProto сложнее, чем через Bot API, и может потребовать custom adapter или отдельный сервис, а не только стандартные ноды n8n.[4][5][6]

### Риск 2. Exchange

В конкретной инфраструктуре Exchange могут быть ограничения по SMTP, IMAP или EWS, поэтому способ интеграции нужно подтвердить на этапе технического обследования.[7][8]

### Риск 3. Качество данных

Если у части людей не хватает телефона, email или Telegram-идентификатора, автоматизация будет неполной.

### Риск 4. Свободный текст

Если пользователи будут отвечать длинными произвольными сообщениями вместо коротких команд, качество распознавания в MVP снизится.

## Политики эскалации и операционные SLA (из интервью)

### Политика напоминаний и эскалаций

- Базовый интервал напоминаний: 1 рабочий день.
- Интервалы `1-е/2-е/3-е` напоминание должны быть настраиваемыми на уровне каждой задачи/шаблона.
- `call-task` создается при выполнении условий: нет ответа по email, не получен/не подтвержден дедлайн, отсутствует результат для проверки.
- Эскалация руководителю выполняется:
  - после второй просрочки дедлайна; или
  - при отсутствии ответа более 4 рабочих дней.
- Тихие периоды: не отправлять в субботу и воскресенье, а также в праздничные дни (по календарю).
- Политики должны поддерживать градации приоритета: `high`, `normal`, `low`.

### SLA и операционная дисциплина

- Максимальный срок разбора нового `call-task` оператором: 2 дня.
- Максимальный срок обработки входящего ответа исполнителя: 2 дня.
- Красная зона очереди оператора: 30 задач (порог должен быть настраиваемым параметром).
- Лимит ручной обработки оператором для планирования нагрузки: до 30 ручных задач в день (звонки/встречи).
- Допустимый процент технических сбоев доставки: до 10%.
- При конфликте целей приоритет отдается скорости реакции, затем качеству проверки `done`.

## Интерфейс приложения (Frontend UX)

### Продуктовые принципы интерфейса

- Frontend должен быть "тонким": UI отвечает за ввод, отображение состояния и запуск действий, а основная бизнес-логика живет в n8n workflow и backend API.
- Структура интерфейса должна поддерживать быстрые изменения без переписывания всего приложения.
- Все ключевые UI-элементы должны собираться из переиспользуемых компонент, а не из одноразовых экранных реализаций.
- Конфигурационные изменения (новые методы верификации, новые типы эскалаций, новые поля policy) должны вноситься преимущественно через схемы/конфиги, а не через hardcode в UI.

### Требования к архитектуре frontend

- Компонентная архитектура: разделение на `atoms` / `molecules` / `organisms` / `pages` (или эквивалент).
- Отдельный слой API-клиента и отдельный слой UI-компонент.
- Использование design tokens (цвета, отступы, типографика, состояния) для единообразного стиля.
- Использование schema-driven форм для сущностей (`person`, `template`, `assignment`, `verification_policy`, `escalation_policy`) там, где это возможно.
- Переиспользуемые таблицы, фильтры, карточки статусов, таймлайн touchpoints и панель действий.

### Как пользователь использует приложение (основной UX flow)

1. Оператор открывает дашборд и видит проблемные задачи: просрочки, задачи без ответа, новые call-task, задачи на verification.
2. Оператор переходит в список assignment, фильтрует по статусу/приоритету/эскалации и выбирает задачу.
3. В карточке assignment оператор видит контекст: дедлайн, историю touchpoints, текущий escalation level, evidence, календарные действия.
4. Оператор выполняет действие из панели quick actions: отправить reminder, эскалировать, назначить звонок, назначить встречу, запустить/повторить verification.
5. Если кейс массовый, оператор запускает batch из шаблона и контролирует прогресс через batch monitor.
6. Администратор настраивает шаблоны, политики эскалации, verification methods и calendar rules.

### Минимальный набор экранов MVP

- `Dashboard`: KPI, счетчики очередей, блоки "критично сегодня", "просрочено", "без ответа", "verification errors".
- `Assignments List`: таблица с фильтрами и сохраненными представлениями.
- `Assignment Details`: карточка задачи + таймлайн + evidence + quick actions.
- `People`: справочник людей и карточка контакта.
- `Projects`: справочник проектов и карточка проекта.
- `Templates & Policies`: шаблоны задач, escalation policy, verification policy.
- `Batch Runs`: запуск массовых кампаний и мониторинг статусов.
- `Campaign Preview & Edit`: предпросмотр и ручная корректировка персонализированных сообщений перед отправкой.
- `Operator Queue`: очереди `call`, `meeting_manual`, `verification_manual`.
- `Settings`: календари, quiet periods, приоритеты, пороги SLA и красной зоны.

- `PriorityBadge` (high/normal/low).

### UX-требования к сценариям проверки результата

- В карточке assignment должно быть видно: какой метод проверки выбран, кто его задал (template/batch/assignment), и через какой sub-workflow выполняется.
- Запуск проверки должен быть доступен вручную кнопкой `Run verification now`, если это разрешено policy.
- UI должен показывать отдельные состояния: "ожидание проверки", "проверка выполняется", "техническая ошибка проверки", "бизнес-проверка не пройдена".
- Для file-based проверки пользователь должен видеть отчет по ошибкам строк и иметь возможность повторной загрузки исправленного файла.

### UX-требования к календарным действиям

- В карточке assignment должна быть кнопка `Find slot` / `Schedule meeting`.
- Пользователь видит предложенные системой слоты (минимум 3) с пояснением по timezone.
- При отсутствии общего слота UI предлагает действие `Create meeting_manual task`.
- После создания встречи UI отображает `meeting_event_id`, время, участников и статус приглашений.

### Роли и права в интерфейсе (MVP)

- Оператор: просмотр/фильтрация assignment, запуск действий, обработка operator queue.
- Администратор: настройка шаблонов, policy, календарей и системных параметров.
- Наблюдатель/руководитель (read-mostly): просмотр статусов, эскалаций и отчетов без изменения policy.

### Нефункциональные UX-требования

- Единый паттерн действий: все ручные действия подтверждаются и пишутся в touchpoint.
- Интерфейс должен оставаться отзывчивым при больших списках (пагинация, серверная фильтрация/сортировка).
- Ошибки API/n8n отображаются человекочитаемо и с техническим кодом для поддержки.
- Локализация дат и времени обязательна; хранение времени в UTC, отображение в выбранной timezone.
- Значения фильтров по умолчанию:
  - `Dashboard`: `overdue`, `no_response`, `verification_manual`, `meeting_manual`, `call`.
  - `Assignments`: `active_only`, `current_project`, `due_in_7_days`.
- Для редактирования задач применяется модель `Revert to previous revision` (версионность), а не классический undo.
- `Retry` в UI обязателен для сценариев `save failed` и `job failed`.

### UX-критика и корректировки (дизайн-аудит)

Ниже зафиксированы критичные UX-риски и обязательные корректировки для MVP.

1. Когнитивная перегрузка карточки assignment:

- Риск: слишком много блоков и действий в одном экране.
- Корректировка: использовать progressive disclosure:
  - базовый слой: статус, дедлайн, следующий шаг, primary action;
  - раскрываемые секции: touchpoints, evidence, calendar, audit;
  - вторичные действия увести в overflow menu.

2. Конкуренция действий в `ActionPanel`:

- Риск: пользователь выбирает неправильное действие (например, эскалация вместо verification).
- Корректировка: в `ActionPanel` показывать только валидные next actions по текущему статусу, остальные disabled с reason tooltip.

3. Недостаточная прозрачность job-based поведения:

- Риск: ощущение "система зависла" при долгих фоновых операциях.
- Корректировка:
  - единый `JobStatusToast/Panel` с прогрессом и ETA (если доступно);
  - статус последней операции закреплять в карточке объекта до terminal state;
  - для long-running jobs показывать "дольше обычного", но без ложной ошибки.

4. Preview & Edit может подорвать доверие:

- Риск: непонятно, что именно будет отправлено и что уже отправлено.
- Корректировка:
  - показывать immutable sent snapshot для отправленных сообщений;
  - четко маркировать editable vs sent items;
  - перед approve показывать итоговый счетчик: `ready/sent/blocked/review-required`.

5. Слабая UX-обработка конфликтов редактирования:

- Риск: потеря изменений между вкладками.
- Корректировка:
  - при `CONFLICT_EDIT` показать diff (local vs latest server revision);
  - предложить действия: `Reload latest`, `Overwrite with comment`, `Create new revision`.

6. Ошибки импорта Excel недостаточно обучающие:

- Риск: оператор не понимает, как исправить данные.
- Корректировка:
  - ошибки показывать не только как список, но и как "actionable hints";
  - скачать CSV с ошибками и колонкой `fix_hint`;
  - всегда показывать блокирующие vs неблокирующие ошибки.

7. Риск "шумных" предупреждений:

- Риск: баннеры и алерты перестают замечаться.
- Корректировка:
  - ввести severity-модель: `info`, `warning`, `critical`;
  - ограничить количество постоянных баннеров на экран;
  - объединять однотипные предупреждения в summary panel.

8. Доступность и читаемость статусов:

- Риск: критичные статусы завязаны только на цвет.
- Корректировка:
  - каждый статус должен иметь иконку/текстовый label, не только цвет;
  - контраст не ниже WCAG AA;
  - клавиатурная навигация обязательна для основных сценариев оператора.

9. Недостаточно пустых и переходных состояний:

- Риск: "пустые таблицы" выглядят как баг.
- Корректировка:
  - определить для каждого ключевого экрана `empty/loading/error/partial` состояния;
  - в empty state давать явное CTA (например, "Загрузить Excel", "Создать шаблон").

10. Информационная архитектура терминов:

- Риск: смешение терминов `Project` и `Target Object`.
- Корректировка:
  - в UI везде использовать явные подписи:
    - `Project (workspace)`;
    - `Target Object (Excel entity)`;
  - добавить tooltip-пояснение в фильтрах и карточках.

### UX Decision Log (зафиксированные решения)

#### Assignment Card: header и структура

- В шапке карточки всегда отображать: `created_at` (дата), `Project`, `card_title`, `deadline`, `responsible_person`.
- Primary action:
  - в списках: `Открыть`;
  - в карточке: `Сохранить`.
- Секции, которые можно сворачивать без потери скорости: `TouchpointTimeline`, `EvidencePanel`, `Audit`, `Raw Payload/Technical`.

#### Матрица next actions (предложение для MVP)

| Статус | Разрешенные действия (2–3) | Disabled (показывать с reason) |
|---|---|---|
| `new` | `Send first message`, `Edit task`, `Cancel` | `Escalate` (нет первичного контакта), `Run verification` (нет DONE-сигнала) |
| `notified` | `Send reminder`, `Edit deadline`, `Escalate` | `Run verification` (ожидается признак выполнения), `Schedule meeting` (ранний этап) |
| `in_progress` | `Send reminder`, `Run verification`, `Update progress` | `Cancel` (если есть активные external checks), `Force escalate` (если не нарушен policy threshold) |
| `done_pending_check` | `Run verification`, `Mark done (manual)`, `Request clarification` | `Escalate` (ожидается финализация проверки), `Schedule meeting` (нецелевой шаг) |
| `overdue` | `Escalate`, `Schedule meeting`, `Create call-task` | `Mark done` (без evidence/check), `Archive` (нет финального исхода) |
| `escalated` | `Next escalation`, `Schedule meeting`, `Create call-task` | `Send first message` (этап уже пройден), `Downgrade level` (без operator reason) |
| `blocked` | `Unblock`, `Request data`, `Set new date` | `Escalate manager` (если блокер валиден и в окне grace-period), `Mark done` (без нового подтверждения) |

#### Опасные действия (double confirmation)

- Для действия `Send message/email` (включая массовую отправку) обязательна двойная проверка:
  - шаг 1: подтверждение действия;
  - шаг 2: итоговый summary (`recipients`, `project`, `ready/blocked/review-required`).

#### Job-based UX

- Отдельная страница/раздел `Jobs` обязательна для наблюдения за фоновыми задачами.
- Порог предупреждения "операция дольше обычного": 120 секунд.
- История job-статусов сохраняется и доступна пользователю 24 часа.

#### Preview & Edit

- Редактируемые поля сообщения: `subject`, `body`, `to`, `attachments`, `cc`.
- После фактической отправки immutable становится весь payload сообщения.
- Минимальные счетчики перед отправкой: `ready`, `sent`, `blocked`, `review_required`.

#### Конфликты редактирования

- При конфликте по одному объекту:
  - сохранение запрещается;
  - действие по умолчанию: `Reload latest`.
- Автолок объекта "кто сейчас редактирует" обязателен.

#### Баннеры и алерты

- Максимум постоянных баннеров на экран: 2.
- `info`-баннеры скрываются автоматически через 30 секунд.
- Классификация `critical` (предложение для MVP):
  - массовая отправка завершилась с системной ошибкой;
  - нарушена целостность данных (`project`/`target_object` mismatch);
  - retry-лимит по каналу исчерпан;
  - невозможность выполнить обязательный verification check;
  - ошибка сохранения, из-за которой есть риск потери пользовательских изменений.

#### Локализация и доступность

- Язык интерфейса и статусов: только русский.
- High-contrast режим в MVP не обязателен.
- Обязательные keyboard-only требования не фиксируются для MVP.

#### Empty/loading/error/partial: критичные экраны и CTA

- Критичные экраны с обязательными empty state:
  - `Dashboard`,
  - `Assignments List`,
  - `Assignment Details`,
  - `Campaign Preview & Edit`,
  - `Operator Queue`.
- CTA для пустого `Campaign Preview & Edit`: `Загрузить Excel/CSV и сгенерировать черновики`.
- Для partial state в рассылках показывать действия:
  - `Продолжить отправку`,
  - `Retry failed`,
  - `Экспорт ошибок`.

#### Термины и фильтры

- Термины в UI:
  - `Project` -> `Проект`,
  - `Target Object` -> `Объект`.
- Приоритет фильтрации: сначала `Проект`, затем `Объект`.
- Onboarding tooltip на первом входе в `Assignments/Campaign` не требуется.

## Definition of Done для MVP

MVP считается готовым, если:

- можно создать человека с обязательными каналами и номером телефона;
- можно создать проект и вести assignment/batch в рамках проекта;
- можно создать шаблон задачи;
- можно запустить batch и получить персональные assignment;
- Telegram client API реально отправляет и принимает сообщения;
- Exchange email реально отправляет и принимает письма;
- scheduler реально поднимает due actions;
- assignment переводится по ключевым статусам;
- для каждой задачи/блока задач можно задать метод проверки результата;
- для внешних систем проверка отрабатывает через sub-workflow и пишет evidence;
- для Excel/CSV проверка отрабатывает через sub-workflow с отчетом ошибок;
- персонализированная рассылка из Excel/CSV проходит этап `Preview & Edit` перед отправкой;
- отправка персонализированной кампании возможна только после явного подтверждения оператором;
- импорт Excel кампании создает `import_version`, фиксирует `imported_at` и `imported_by`;
- при не найденном owner создается review-задача на дополнение справочника людей;
- при падении каналов оператор может вручную проставить `email_sent_flag` с комментарием;
- backend-API является отдельным сервисом и владельцем state machine;
- для Telegram/Email/API действуют retry до 3 попыток с backoff;
- при просрочке создается call-task;
- при срабатывании meeting escalation система подбирает слот и создает встречу в календаре;
- можно увидеть историю touchpoints по задаче;
- оператор получает daily digest;
- календарь рабочих/нерабочих дней влияет на планирование напоминаний и эскалаций;
- хотя бы один пилотный бизнес-сценарий отрабатывается end-to-end.

## Технические контракты MVP (v1, зафиксированные решения)

### API и протокол

- Базовый префикс API: `/api/v1`.
- Авторизация в MVP: `Bearer token`.
- Для тяжелых операций используется job-based паттерн:
  - API отвечает `202 Accepted` + `job_id`.
- Endpoint отмены job обязателен: `POST /jobs/{id}/cancel`.
- Дефолтный `page_size` для списков (`assignments`, `jobs`, `campaigns`): `50`.
- Формат дат/времени: `ISO-8601 UTC` (пример: `2026-04-27T11:00:00Z`).

### Модель данных

- Подтвержденные таблицы MVP:
  - `projects`,
  - `target_objects`,
  - `task_assignments`,
  - `task_batches`,
  - `campaigns`,
  - `campaign_messages`,
  - `touchpoints`,
  - `status_history`,
  - `evidence`,
  - `operator_queue`,
  - `jobs`,
  - `revisions`,
  - `imports`.
- Для assignment применяется отдельный `external_key` (детерминирующий бизнес-ключ).
- Для автолока редактирования используются поля:
  - `locked_by`,
  - `locked_at`,
  - `lock_expires_at`.

### Статусы и правила переходов

- Финальные terminal statuses:
  - `done`,
  - `cannot_be_done`,
  - `cancelled`.
- `cannot_be_done` — отдельный статус (не вариант `cancelled`).
- В `cannot_be_done` может переводить любой пользователь с правом изменения задачи.
- Автопереход `overdue -> escalated` выполняется только при `policy=true`.
- Для статуса `blocked` действует автоэскалация через 2 недели.

### Excel import contract

- Поддерживаемые колонки v1 (точные названия):
  - `Дирекция`,
  - `Проект`,
  - `Статус КТ`,
  - `Количество КТ`,
  - `Количество этапов`,
  - `РП`,
  - `Куратор`,
  - `Последнее изменение`,
  - `Ссылка КТ`,
  - `Ссылка на проект`,
  - `Статус проекта в КТ`,
  - `Статус согласования`.
- Минимально обязательные поля:
  - `target_object_external_key`,
  - `target_object_name`,
  - `responsible_email`.
- Дубли по `target_object_external_key` считаются ошибкой импорта.
- Пустой `responsible` блокирует всю загрузку.
- Режим `dry run` обязателен.
- Ограничение размера файла для MVP явно не вводится.

### Error / Job contract

- Обязательные поля ошибки:
  - `code`,
  - `message`,
  - `details`,
  - `retryable`,
  - `correlation_id`.
- Локализация текстов ошибок: только RU.
- Terminal статусы job:
  - `succeeded`,
  - `failed`,
  - `timed_out`,
  - `cancelled`.
- `timed_out` по умолчанию: 1 час.
- Для `failed` сохраняются:
  - `last_retry_at`,
  - `retry_count`.

### RBAC и аудит

- Роли MVP:
  - `operator`,
  - `admin`,
  - `viewer`.
- `operator` может и выполнять задачи, и менять policy.
- `admin` может делать manual fallback (`email_sent_flag`).
- `viewer` видит PII (`email`, `phone`) без маскирования.
- Audit "кто сделал действие" обязателен для всех mutating endpoint.

### Интеграции

- Exchange отправка: от персонального аккаунта оператора.
- Calendar встречи: всегда от имени оператора.
- Входящие Telegram-сообщения не связываются автоматически по `task_code`.
- Входящее сообщение без привязки к задаче отправляется в manual queue.
- Rate limit на исходящие сообщения в MVP не задается.

### KPI и аналитика

- KPI исхода:
  - `(done + cannot_be_done) / created`.
- KPI времени:
  - `median(created_at -> final_outcome_at)`.
- Окно полезности эскалации: `N = 7 дней`.
- KPI по умолчанию считаются по всем проектам.

### Frontend технический стек

- `React` + `Next.js`/`Vite` семейство подтверждено.
- Режим рендеринга: `SSR`.
- UI library: `MUI`.
- Feature flags для rollout экранов: обязательны.
- Стратегия data-layer: `React Query`.

## Backend Implementation Decisions v1

### API, идемпотентность и jobs

- Все mutating endpoint обязаны принимать `Idempotency-Key` в заголовке.
- TTL для результата идемпотентности: 24 часа.
- Повтор с тем же `Idempotency-Key` и другим payload возвращает `409 Conflict`.
- Для job-based операций API возвращает `202 Accepted` + `job_id`.
- В ответах job-based операций обязателен `Location` header на `/jobs/{id}`.
- Endpoint отмены job обязателен: `POST /jobs/{id}/cancel`.
- Формат дат/времени: ISO-8601 UTC.
- Дефолтный `page_size` для списков: 50.

### Assignment key и импорт

- `assignment.external_key` является отдельным стабильным бизнес-ключом.
- Рекомендуемая формула внешнего ключа:
  - `hash(project_id + target_object_external_key + template_id + assignment_kind)`.
- `external_key` должен быть стабильным между импортами Excel.
- Изменение `responsible` обновляет существующий assignment, а не создает новый.
- Импорт выполняется атомарно: при ошибках в части строк загрузка не применяется целиком.
- `dry run` возвращает ошибки и diff (`create/update/no-change`).
- `imports.status` обязателен (`draft/validated/applied/failed`).
- Применить import может любая роль с правом изменения данных (в текущем MVP — все роли).
- Если объект исчез из новой версии Excel, задача остается активной и помечается warning.
- `target_objects.last_seen_in_import_version` хранится и обновляется.
- Изменения атрибутов объекта из Excel (например, статус/куратор) обновляются автоматически.

### Состояния, конкуренция и блокировки

- Terminal statuses: `done`, `cannot_be_done`, `cancelled`.
- `cannot_be_done` — отдельный статус.
- Перевод в `cannot_be_done` допустим вручную.
- Автопереход `overdue -> escalated` срабатывает только если policy это разрешает.
- Для `blocked` применяется автоэскалация через 2 недели.
- Разрешен manual override переходов статусов, ограниченных policy.
- Для PATCH обязателен optimistic lock через `revision`.
- При несовпадении `revision` возвращается `409 CONFLICT_EDIT`.
- При одновременных действиях по одной задаче применяется правило first-write-wins.

### Retry, scheduler и очереди

- Автоматический retry: 3 попытки с фиксированным backoff.
- После исчерпания автопопыток доступен ручной `Retry`.
- Для защиты от retry storm рекомендуется jitter на backoff (например, +/-20%).
- При `cancel job` требуется компенсирующий откат побочных эффектов.
- Scheduler interval — configurable (базовый пресет: 10 минут).
- Для параллельных worker используется `SELECT ... FOR UPDATE SKIP LOCKED`.
- Для восстановления после падения worker используется visibility timeout / lease-модель:
  - задача берется в обработку с `lease_until`,
  - при истечении lease может быть переобработана другим worker.
- Лимит одновременных исходящих отправок в MVP не задается.

### Каналы и интеграции

- Email считается отправленным по факту SMTP accepted.
- Для manual `email_sent_flag` отдельный обязательный reason не требуется.
- Telegram inbound без привязки к задаче создает `operator_queue` item типа `inbound_unmatched`.
- Встречи в календаре создаются/обновляются от имени оператора.
- При переносе встречи обновляется существующий `meeting_event_id`.

### Verification и evidence

- Для `sql_query` / `http_api` / `file` действует единый контракт результата:
  - `verification_status`,
  - `business_outcome`,
  - `technical_error_code`,
  - `evidence_ref`.
- `cannot_be_done` по спорным verification-кейсам выставляется вручную.

### Аудит, наблюдаемость и качество данных

- Аудит пишется асинхронно.
- Аудит должен включать diff по полям, а не только "кто/что/когда".
- Audit log является append-only.
- `viewer` видит PII (`email`, `phone`) в рамках текущей модели доступа.
- `correlation_id` генерируется на gateway и передается по всей цепочке.
- Метрики обязательны:
  - import fail rate,
  - job timeout rate,
  - escalation effectiveness.
- Логи сервисов обязательны в structured JSON.

### Error semantics (warning vs error)

- `error` используется, когда операция не может быть корректно завершена и требует повтора/исправления:
  - `409 CONFLICT_EDIT`,
  - `422 IMPORT_VALIDATION_FAILED`,
  - `503 INTEGRATION_UNAVAILABLE`.
- `warning` используется, когда операция завершена, но есть важные оговорки:
  - partial send (частичный успех),
  - объект отсутствует в новой версии Excel, но задача активна,
  - применен manual fallback.

### Масштаб пилота и производительность

- Верхняя граница пилота: до 1000 assignment.
- Целевой `p95` response time для списков и карточек: до 60 секунд.

## Retention + Freshness SLA (Final)

### 1) Retention policy для evidence

- `evidence.metadata` хранится бессрочно.
- `evidence.payload` хранится 180 дней, затем переводится в архив/удаляется по retention policy.
- После удаления payload в системе сохраняются ссылки на evidence, технический итог проверки и audit trail.

Статус: **зафиксировано**.

### 2) Freshness SLA (eventual consistency)

- Целевой freshness SLA для UI: до 60 секунд.
- Hard max lag для пользовательских сценариев: до 120 секунд.
- Job-based операции должны отображать промежуточный результат сразу через polling/status widgets.

Статус: **зафиксировано**.
